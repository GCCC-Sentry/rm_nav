# 全向感知改动指南

本文档面向第一次接手这套代码的人，目标是回答下面几个问题：

- `sentry_omni26` 和 `sp26/sp_vision_25` 分别负责什么
- 为什么不能直接把全向感知“塞一个 yaw 变量”进主瞄
- 这次到底改了哪些文件
- 双 yaw 车上，坐标到底怎么处理
- `auto_aim_debug_mpc` 和 `sentry_multithread` 各自现在能做什么
- 从零开始，怎么把 omni26 融到 sp26，并最终跑通

本文尽量把这次改动背后的逻辑、实际踩过的坑、具体命令和验证方式都写清楚。

## 1. 项目结构和角色分工

当前相关代码有两套：

- 主瞄 / 哨兵控制：
  - `/home/asus/sp26/sp_vision_25`
- 全向感知：
  - `/home/asus/sentry_omni26`

两套代码的职责不同。

### 1.1 `sp26/sp_vision_25`

这是主瞄和整体控制主程序，负责：

- 输出融合目标 `/enemy_targets`

它不应该直接负责主瞄精瞄或射击控制。它更像一个“上游感知节点”。

## 2. 为什么不能只统一一个 yaw

最开始很容易想到一种做法：

- 主瞄给一个 `yaw`
- 全向感知也给一个 `yaw`
- 状态机判断当前用谁

这个思路不够，因为：

- `yaw` 本身没有目标语义
  - 不知道是哪个目标
  - 不知道优先级
  - 不知道是否连续跟踪
- 没有 `track_id`
  - 会导致同优先级多目标反复切换
- 没有 `confidence`
  - 状态机没法稳定决策
- 没有 `x/y/z`
  - pitch 无法一起正确生成
- 主瞄和全向都可能“看到了”，但不能只靠两个角度硬覆盖

所以正确的做法是：

- 统一的是“目标语义层”
- 最终只输出一个 `io::Command`
- 仲裁发生在状态机，不发生在某个裸 `yaw` 变量上

也就是：

```text
主瞄检测 -> 主瞄目标
全向融合 -> 融合目标
            \ 
             -> 状态机仲裁 -> 最终 io::Command -> 下位机
            /
主瞄跟踪 -> 主瞄精瞄命令
```

## 3. 两套程序原来的差异

### 3.1 `sp26` 原来的“全向”是什么

`sp26` 早期的全向逻辑并不是融合目标，而是：

- 多路 USB 相机各自检测
- 生成单相机 `DetectionResult`
- 再按优先级排序

它的语义更接近：

- “某一路相机看到一个目标，建议朝某个角度转”

对应的旧文件主要有：

- `tasks/omniperception/perceptron.*`
- `tasks/omniperception/decider.*`

### 3.2 `omni26` 的输出是什么

`omni26` 输出的是融合后的目标：

- 目标编号
- 优先级
- 跟踪状态
- 目标位置 `x/y/z`
- 方位角 `yaw`
- 置信度
- `track_id`

它的语义更接近：

- “已经在多相机上融合好的全局候选目标”

所以两者不能直接用同一个数据结构。

## 4. 正确的集成原则

本次采用的原则是：

1. `omni26` 保持独立运行
2. `sp26` 通过 ROS2 订阅 `/enemy_targets`
3. `sp26` 内部新建专门的数据抽象承接 omni 目标
4. 状态机决定当前是主瞄精瞄还是 omni 引导
5. 最终只发一个 `io::Command`

不采用的方案：

- 直接把 `omni26` 整套 detector / combine / camera 线程拷进 `sp26`
- 只做一个统一 `yaw` 变量

原因：

- 前者耦合太重，维护困难
- 后者表达力不够，现场容易抖

## 6. 双 yaw 车为什么是难点

你的车结构是：

- 主瞄在小 yaw 上
- 全向感知 3 相机在大 yaw 上

这就意味着：

- omni 看到的目标角，天然是“大 yaw 参考系”
- 主瞄和下位机真正要用的角，天然是“小 yaw 参考系”

如果不补偿，结果会错。

### 6.1 初始直线对齐假设

这次采用的工程假设是：

- 比赛刚开始时，大 yaw 和小 yaw 在同一直线上

在这个前提下：

- 主瞄 IMU 当前的 yaw 相对初始零位变化量
- 可以近似看成 `small_yaw_relative_big`

于是可得：

```text
yaw_target_small = yaw_target_big - yaw_small_relative_big
```

同时不仅要补偿 `yaw`，还要补偿坐标：

```text
x_small = cos(delta) * x_big + sin(delta) * y_big
y_small = -sin(delta) * x_big + cos(delta) * y_big
z_small = z_big
yaw_small = yaw_big - delta
```

这里：

- `delta = small_yaw_relative_big`

## 7. 为什么一开始改到 `omni_yaw_bridge_node` 还不够

一开始做过一个版本：

- 在 `omni26` 的 `omni_yaw_bridge_node` 里订阅 `/small_yaw_angle`
- 算出补偿后的 `/omni_yaw_angle`

这个改动本身没有错，但路径不对。

因为真正控制链不是：

```text
/omni_yaw_angle -> 下位机
```

而是：

```text
/enemy_targets
  -> SubscribeOmniTargets
  -> FusionCue
  -> Tracker / OmniCommandBuilder
  -> io::Command
  -> 下位机
```

所以如果只补 `/omni_yaw_angle`，但 `sp26` 真正控枪的地方仍然直接用 `/enemy_targets`，那补偿就没进控制链。

因此最终正确做法是：

- 把双 yaw 补偿前移到 `SubscribeOmniTargets`
- 让进入 `FusionCue` 的目标就已经是小 yaw 系

## 9. `SubscribeOmniTargets` 是这次最关键的接点

文件：

- `/home/asus/sp26/sp_vision_25/io/ros2/subscribe_omni_targets.hpp`
- `/home/asus/sp26/sp_vision_25/io/ros2/subscribe_omni_targets.cpp`

### 9.1 它现在做了什么

它现在不只是订阅 `/enemy_targets`，还会：

- 订阅 `/small_yaw_angle`
- 维护最新的 `small_yaw_relative_big`
- 收到 `/enemy_targets` 后，立即把目标从大 yaw 系旋到小 yaw 系
- 再生成 `FusionCue`

### 9.2 为什么这一步必须放在这里

因为 `FusionCue` 是 `sp26` 内部全向目标的统一抽象。

只要保证：

- 进入 `FusionCue` 的东西已经是小 yaw 系

那后面的：

- `Tracker`
- `OmniCommandBuilder`
- `auto_aim_debug_mpc`
- `sentry_multithread`

都不需要再各自重复做补偿。

这能避免：

- 某个地方忘了补
- 一个地方补一次，另一个地方又补一次

### 9.3 现在的处理逻辑

当前逻辑大致是：

1. 收到 `/small_yaw_angle`
   - 保存当前小 yaw 相对初始零位的偏角
2. 收到 `/enemy_targets`
   - 用这个偏角旋转 `x/y`
   - 用这个偏角修正 `yaw`
3. 结果写入 `FusionCue`
4. 按优先级、跟踪状态、置信度排序

## 10. `Tracker` 是怎么接 omni 的

文件：

- `/home/asus/sp26/sp_vision_25/tasks/auto_aim/tracker.hpp`
- `/home/asus/sp26/sp_vision_25/tasks/auto_aim/tracker.cpp`

新增了一个重载：

- `track(const std::vector<FusionCue>& omni_targets, std::list<Armor>& armors, time_point t)`

它的作用不是让 omni 直接打，而是：

- 在状态机里把 omni 作为“可切换候选目标”

核心思想：

- 主瞄稳定 `tracking` 时，仍然主瞄优先
- omni 只在需要切换时提供候选目标

大致逻辑：

- `lost`
  - 主瞄如果没目标，返回空主目标
  - 上层可使用 omni 引导
- `tracking`
  - 如果主相机画面内出现更高优先级目标，优先切主相机目标
  - 如果 omni 出现更高优先级目标且当前主目标已收敛，进入 `switching`
- `switching`
  - 等主相机真正看见该目标，再切回主瞄链

这比“谁发的 yaw 新就听谁的”稳定得多。

## 11. `sentry_multithread` 现在怎么工作

文件：

- `/home/asus/sp26/sp_vision_25/src/sentry_multithread.cpp`

### 11.1 旧逻辑

没有 `sentry_interfaces` 时：

- 仍走旧 `Perceptron`
- 保持原有兼容路径

### 11.2 新逻辑

有 `sentry_interfaces` 时：

- 创建 `SubscribeOmniTargets`
- 启动 ROS2 spin 线程
- 主循环里读取 `omni_targets`
- 调 `tracker.track(omni_targets, armors, timestamp)`

然后按状态决定命令来源：

- `tracking / detecting / temp_lost`
  - 用主瞄 `Aimer`
- `lost / switching`
  - 用 `OmniCommandBuilder`

### 11.3 `/small_yaw_angle`

`sentry_multithread` 还会持续发布：

- `/small_yaw_angle`

方式是：

- 启动时取当前云台 yaw 为零位
- 后续发布 `current_yaw - zero_yaw`

这样 omni 目标进入 `sp26` 时就能做双 yaw 补偿。

## 12. `auto_aim_debug_mpc` 现在怎么工作

文件：

- `/home/asus/sp26/sp_vision_25/src/auto_aim_debug_mpc.cpp`

这是这次用户特别关心的一点：

- “我想启动 `auto_aim_debug_mpc` 的时候也有全向感知”

现在已经实现了。

### 12.1 之前的问题

之前 `auto_aim_debug_mpc` 只做：

- 主瞄检测
- 跟踪
- MPC 规划
- 发给云台

它不消费 `/enemy_targets`，所以：

- omni 看见了也没反应

### 12.2 现在的逻辑

现在 `auto_aim_debug_mpc` 会：

1. 启动 `SubscribeOmniTargets`
2. 持续发布 `/small_yaw_angle`
3. 保持原有主瞄 MPC 线程
4. 额外维护一个 `omni_command_override`

仲裁逻辑是：

- 主瞄有目标
  - 继续走主瞄 MPC
- 主瞄无目标，但 omni 有目标
  - 用 `OmniCommandBuilder` 生成引导命令
  - 由规划线程直接下发
  - `shoot=false`

也就是说现在的 `auto_aim_debug_mpc` 是：

- 主瞄优先
- 丢失时 omni 引导找目标

它仍然不是完整哨兵状态机，但已经具备你最需要的能力。

## 22. 当前版本的能力边界

现在这版已经能做到：

- `sentry_multithread` 使用 omni26 的融合目标
- `auto_aim_debug_mpc` 也能在主瞄丢失时使用 omni 引导
- 双 yaw 补偿进入了真正控制链
- 主瞄和 omni 能共用 `/small_yaw_angle`

但仍然要明确几点：

### 22.1 omni 当前仍然是“引导转头”，不是“直接射击”

`OmniCommandBuilder` 当前：

- `shoot = false`
- `pitch` 也是粗算

这是有意设计，不是缺陷。

### 22.2 `auto_aim_debug_mpc` 仍不是完整哨兵状态机

它现在是：

- 主瞄优先
- 主瞄无目标时 omni 引导

而不是完整的：

- 高优先级切换
- 主瞄重锁
- 更复杂状态仲裁

完整状态机还是以 `sentry_multithread` 为主。

### 22.3 双 yaw 零位依赖初始对齐

当前补偿成立依赖：

- 启动时大 yaw 和小 yaw 共线

如果这个前提不满足，误差会直接传到 omni 引导角。
